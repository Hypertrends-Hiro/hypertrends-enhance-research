Telemetry Robust Plan: Catalog + SystemCatalogConfig

Plan maestro para evolucionar telemetry a una plataforma robusta con gobierno de eventos, control operativo por sistema y enrutamiento multidestino (Braze, GA4 y futuros destinos).

Catalog canónico Config por sistema PascalCase oficial Admin vía API (Postman)

Lista de tareas ejecutables: tasks.html

1) Objetivo

  1. Catalog: diccionario global de eventos (nombres canónicos + metadatos).
  2. SystemCatalogConfig: reglas operativas por sistema/tenant (enable, routing, mappers).

Esta separación evita mezclar semántica del evento con reglas de operación por aplicación.

2) Principios de diseño

  • Separación de responsabilidades: catálogo define “qué existe”; config define “cómo opera”.
  • Nombre canónico único: todos los nuevos eventos en PascalCase.
  • Ingest gobernado por datos: nada de reglas hardcodeadas.
  • Trazabilidad total: toda decisión de ingest queda auditada.
  • Compatibilidad evolutiva: aliases legacy permitidos durante migración.

3) Alcance funcional

Incluye

  • API de Catalog y SystemCatalogConfig.
  • Validación en POST /api/v1/telemetry/ingest y /ingest/batch.
  • Resolución de nombres (canónico + aliases).
  • Routing por destino (Braze/GA4) por configuración.
  • Administración solo con HTTP: endpoints para alta en catálogo y para configuración por sistema (Postman, Insomnia, curl). Sin front dedicado.

No incluye (fase futura)

  • Front de administración (React/Vue/etc.): el operador usa Postman u otro cliente HTTP.
  • Analytics UI avanzada.
  • Workflow complejo de aprobaciones.
  • Arquitectura multi-región active-active.

4) Seed de catálogo: playground normalizado (JSON)

Fuente única de listas: telemetry/.plan/braze-sdk-playground.html (SDK_EVENT_NAMES + TELEMETRY_EVENT_NAMES). Se genera un contrato máquina-legible con nombres en PascalCase, aliases, columnas donde apareció cada nombre, etiquetas de dominio (UX, negocio, clínica, commerce, identidad, etc.) y pistas de forma para Braze vs GA4.

  • JSON: telemetry/.plan/playground-events-catalog.normalized.json
  • Regenerar: python telemetry/scripts/gen_playground_events_catalog.py (desde la raíz del repo; o python scripts/gen_playground_events_catalog.py desde telemetry/).

Cada ítem en events[] incluye canonical_name, aliases, domain_tags[], braze.primary_surface (custom event, meta-standard name, purchase signal, user attributes patch), braze.rest_track_arrays_typically_used (events / purchases / attributes), ga4.measurement_protocol_shape_hint, y si el nombre es de la familia “standard” del playground (meta_standard_event_name). Caso extra: user_profile_patch → UserProfilePatch (solo en JS del playground, no en el array telemetry).

Braze (primary_surface) Significado
custom_event_domain Evento de producto; REST events[] / Web logCustomEvent.
custom_event_meta_standard_name Mismo transporte; nombre alineado a etiquetas tipo “standard” (AddToCart, PageView…).
purchase_revenue_signal Compra/ingreso; el mapper puede usar purchases[] y/o evento con revenue.
user_attributes_patch Perfil/atributos; REST attributes[] como superficie principal; evento opcional para auditoría.

Uso en implementación: Fase 0 importa este JSON a tablas catalog_events + catalog_event_aliases; la administración por API y SystemCatalogConfig referencian siempre canonical_name. Los emisores pueden seguir enviando strings legacy hasta migrar; el ingest resuelve alias → canónico.

5) Modelo conceptual

5.1 Catalog (global)

  • event_key (UUID)
  • canonical_name (único, PascalCase)
  • display_name, description, domain
  • lifecycle_status: active|deprecated|retired
  • aliases (opcional, transición)
  • Auditoría: created_at/by, updated_at/by

5.2 SystemCatalogConfig (por sistema)

  • config_id (UUID), system_id, tenant
  • event_key (FK a Catalog)
  • enabled, mode (allow|deny|shadow)
  • destinations: braze/ga4/warehouse
  • mappers: braze.mapper_version, ga4.mapper_version
  • validation_policy, vigencia, auditoría

5.3 Ingest audit trail

  • message_id, original_event_name, canonical_event_name
  • system_id, tenant, source_channel
  • decision: forwarded|accepted_not_forwarded|rejected
  • decision_reason, resultados por destino y timestamps

6) Contratos API propuestos

Alcance acordado para administración: solo APIs, consumibles con Postman (u otro cliente HTTP). No se incluye aplicación front ni paneles; toda la operación de alta de eventos y de configuración vive en estos endpoints.

6.1 MVP administración (Postman)

Área Endpoints (mínimo viable)
Catálogo — nuevos eventos POST /api/v1/catalog/events — crear evento canónico (PascalCase), metadatos y aliases opcionales en el body.
Opcional para operar con comodidad: GET /api/v1/catalog/events y GET /api/v1/catalog/events/{canonical_name} (solo lectura, sin UI).
Configuración por sistema POST /api/v1/system-catalog-configs — crear regla (sistema, tenant, evento, enabled, destinations, mappers).
PATCH /api/v1/system-catalog-configs/{config_id} — actualizar toggles y routing sin redeploy.
Opcional: GET /api/v1/system-catalog-configs con filtros (system_id, tenant, evento) para inspección en Postman.

6.2 Ampliación futura (no bloquea MVP)

Si más adelante hace falta gobierno fino sin scripts: PATCH de catálogo, DELETE de aliases, activate/deactivate explícitos, etc.

Módulo Endpoints adicionales (referencia)
Catalog PATCH /api/v1/catalog/events/{canonical_name}, POST/DELETE …/aliases
Support POST /api/v1/catalog/resolve — probar resolución alias → canónico (QA).

7) Reglas de decisión en ingest

  1. Validar envelope base (meta, event, message_id, source_system).
  2. Resolver event.name (canónico o alias).
  3. Buscar config por (system_id, tenant, event_key).
  4. Aplicar política (enabled, mode, destinos).
  5. Transformar por destino: canónico -> Braze y canónico -> GA4.
  6. Enviar, auditar y responder estado trazable.

8) Política de naming

  • Oficial: eventos nuevos en PascalCase.
  • Evitar prefijos técnicos en canónico (backend_...).
  • Origen técnico via metadata (source_system, source_channel).
  • Aliases legacy con fecha de retiro.

Ejemplos: intake_incomplete -> IntakeIncomplete, subscription_cancelled|subscription_canceled -> SubscriptionCanceled.

9) Administración vía API (sin front)

No hay producto UI de backoffice en el alcance. Operaciones y “toggles” se hacen con Postman (o curl, Insomnia, etc.) contra los endpoints de la sección 6.

Qué cubre

  • Catálogo: dar de alta nuevos eventos canónicos (POST /catalog/events).
  • Configuración: crear y ajustar reglas por sistema/tenant/evento (POST / PATCH en system-catalog-configs) — ahí vive enable/disable y destinos Braze/GA4.
  • Consulta: si hace falta verificar, usar los GET opcionales documentados en 6.1.

Seguridad y trazabilidad

  • API keys o auth de servicio en los endpoints de administración (no públicos como ingest en fase inicial si aplica).
  • Auditoría en servidor de quién cambió qué (cabecera de actor, API key id, o integración con IdP más adelante).
  • Monitoreo operativo (logs/métricas de ingest) sustituye pantallas de “operations” hasta que exista UI.

10) Payload por destino

Braze y GA4 deben recibir payloads finales distintos. El envelope canónico es único, la transformación es por destino.

  • canonical → braze: cuerpo sparse para /users/track — solo claves con datos (events, y si aplica purchases y/o attributes).
  • canonical → ga4: Measurement Protocol — events[] siempre que haya hit; compras como evento purchase + params ecommerce; perfil como user_properties / user_id cuando el mapper lo defina.

Regla de emisor: para un simple click no envíes bloques vacíos; omití purchase / user.attributes salvo que el contrato de ingest exija un default explícito (purchase.enabled: false).

11) Plan de implementación: Braze + GA4 gobernados por config

Objetivo: una sola ingesta, dos (o más) destinos opcionales, cada uno encendido o apagado por SystemCatalogConfig, con mappers versionados y payloads de salida mínimos (sin arrays vacíos).

11.1 Forma de SystemCatalogConfig.destinations (propuesta)

{
  "braze": {
    "enabled": true,
    "mapper_version": "braze_track_v1",
    "options": { "endpoint_env": "BRAZE_REST_ENDPOINT" }
  },
  "ga4": {
    "enabled": true,
    "mapper_version": "ga4_mp_v1",
    "options": {
      "measurement_id": "G-XXXX",   // o por tenant / secret
      "api_secret_ref": "GA4_MP_SECRET_KWT"
    }
  }
}

11.2 Componentes a implementar

  1. Resolver: event.name → canonical_name (alias table + case rules).
  2. Policy loader: lee config por (system_id, tenant, canonical_name) con cache TTL (ej. Redis o in-process + invalidación).
  3. Dispatcher: si enabled global del evento es falso → auditar y cortar; si true → iterar destinos con destinations.*.enabled.
  4. Mapper registry: braze_track_v1, ga4_mp_v1 — entrada: envelope + catalog row (primary_surface hints) + config options; salida: payload listo para HTTP.
  5. Braze client: construye JSON sparse; solo incluye events, purchases, attributes si el mapper devolvió contenido no vacío.
  6. GA4 MP client: un request puede llevar events + opcionalmente user_properties; compras = evento purchase con params estándar.
  7. Audit store: por message_id: decisión, destinos intentados, HTTP status / error corto por destino, timestamps.
  8. Respuesta al cliente: agregado (ej. destinations: { braze: "ok 201", ga4: "skipped_disabled" }) sin filtrar secretos.

11.3 Fases técnicas (dual destination)

  • Fase A: Modelo DB + CRUD config; seed Braze como hoy; feature flag ga4.enabled default off.
  • Fase B: Integrar resolver + policy en /ingest; Braze detrás de config (no enviar si destinations.braze.enabled es false).
  • Fase C: Mapper GA4 v1 + cliente MP + secretos por entorno; métricas por destino.
  • Fase D (robustez): outbox + worker + reintentos por destino; dead-letter si falla permanente.

12) Roadmap

Fase 0 (2-3 días)

  • Tablas base y validadores PascalCase.
  • Seed desde playground-events-catalog.normalized.json (regenerable con el script).

Fase 1 (4-6 días)

  • APIs mínimas: POST /catalog/events + POST/PATCH (y GET opcional) de system-catalog-configs.
  • Auth en rutas de admin + auditoría de cambios; documentar colección Postman.

Fase 2 (4-6 días)

  • Ingest gobernado por catálogo/config (incl. toggles destinations.braze / destinations.ga4).
  • Estados estándar de respuesta y notas por destino.

Fase 3-4 (8-13 días)

  • Mappers versionados Braze + GA4 MP; payloads sparse; secretos por entorno/tenant.
  • Outbox/reintentos opcional; endpoints admin estables y colección Postman mantenida.

13) Criterios de aceptación

  1. Solo se envía a destino si existe en catálogo y está habilitado en config de sistema.
  2. Naming canónico PascalCase + resolución por aliases legacy.
  3. Toda ingest deja rastro auditable de decisión.
  4. Activar/desactivar y routing por destino se hace vía API de configuración, sin despliegue ni UI propia.
  5. Braze y GA4 reciben payload transformado según config vigente (cada uno puede estar off sin afectar al otro).
  6. Salida a Braze/GA4 es sparse: no se envían claves o arrays vacíos “por completar el contrato”.

  4. Ejecutar Fase 0 y Fase 1 en el próximo sprint.
