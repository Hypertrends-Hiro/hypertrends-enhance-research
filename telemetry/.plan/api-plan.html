Plan API (payload base Braze-generico)

Objetivo: que todos los subsistemas envien un payload base unico y luego inyecten su payload funcional, mientras el backend convierte 1:1 a formato Braze.

Documentacion de uso

  • telemetry/docs/API.md — endpoints implementados, curl, Fase 1 (sin auth).
  • braze-to-telemetry-migration.html — evento por evento: codigo Braze actual → body POST /ingest.
  • braze-sdk-playground.html — pagina local con Braze Web SDK: botones UX vs backend-style (solo pruebas; no subir API keys).
  • payload-usage.html — contrato universal: occurred_at, message_id, duplicados, sesion, atributos, ejemplos.
  • frontend-usage.html — checklist y ejemplos para SPAs / navegador.
  • backend-usage.html — servicios, crons, scripts, compras y adapters legacy.
  • all-events.html — inventario de eventos detectados en el workspace.
  • events.md — split backend / frontend UX / sesion (markdown).

5) Reglas obligatorias

  • Siempre enviar meta.message_id para idempotencia.
  • event.name debe existir en catalogo oficial.
  • El backend decide si rellena attributes, events, purchases para Braze.
  • Si purchase.enabled=true, se construye objeto de compra Braze.
  • Campos PII solo en user; no duplicar PII en event.properties.

Este contrato permite que cualquier subsistema componga el mismo envelope y solo cambie su payload funcional.
