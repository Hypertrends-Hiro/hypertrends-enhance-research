Documentacion de uso: payload universal

Esta guia explica por que existen occurred_at, message_id, datos de sesion y como usar el payload base sin obligar a todos los subsistemas a llenar todo.

1) Respuesta corta

  • No: no todos los subsistemas envian todo el payload.
  • Si: todos envian la misma estructura base para mantener compatibilidad.
  • Required minimo: meta.message_id, meta.occurred_at, meta.source_system, event.name, event.time.

2) Por que existe occurred_at?

occurred_at es el momento en que el evento ocurrio en el sistema origen, no el momento en que lo recibio el API.

EscenarioSin occurred_atCon occurred_at
Cola/reintentoEl evento parece mas reciente de lo realSe conserva la hora real de negocio
Batch nocturnoSe distorsiona el funnel por horaSe mantiene orden cronologico correcto
Latency / SLANo puedes medir ingest delayPuedes medir received_at - occurred_at

New Date se usa, pero en el origen del evento. Si el frontend dispara el evento, hace new Date().toISOString() en ese instante y lo manda en occurred_at.

3) Por que existe message_id?

Para idempotencia. Si el mismo evento se reenvia (retry, timeout, doble click, red inestable), el backend no lo cuenta dos veces.

CasoSin message_idCon message_id
Retry automaticoDuplicas evento en BrazeEl segundo intento se ignora como duplicado
Doble submitConversion infladaUna sola escritura valida
Reproceso batchRiesgo alto de doble contabilizacionReproceso seguro

Quien lo genera? El subsistema origen. Recomendado: UUIDv7.

El otro sistema debe guardarlo? Idealmente si, para trazabilidad. Pero el requisito minimo es que lo envie; el servicio telemetry lo persiste y deduplica.

Que pasa si se duplica? Si llega el mismo message_id + mismo tenant, el API responde 200/202 con estado duplicate_ignored (no error operativo) y no vuelve a publicar a Braze.

4) Datos de sesion: cuando si, cuando no

CampoFrontend webBackend/cronUso
session.session_idRequiredOptionalUnir eventos de una misma sesion
session.page_urlRecommendedOptionalContexto de navegacion
session.routeRecommendedOptionalAnalitica por vista
session.user_agentOptionalOptionalDebug de cliente/dispositivo
session.ipOptional/MaskedOptional/MaskedSeguridad/fraude (segun compliance)

No mandes PII innecesaria en session. Si se requiere IP, aplicar mascara/retencion corta segun politica.

5) Contrato universal: Required / Conditional / Optional

PathNivelNotas
meta.message_idRequiredUUIDv7 recomendado, base de idempotencia
meta.occurred_atRequiredHora real de ocurrencia del evento
meta.source_systemRequiredOrigen: app/sistema emisor
event.nameRequiredDebe existir en catalogo
event.timeRequiredNormalmente igual a occurred_at
user.external_id o user.anonymous_idConditionalAl menos uno
event.propertiesConditionalSegun el evento
purchase.*ConditionalSolo si purchase.enabled=true
session.*Optional/RecommendedEspecialmente importante en frontend
user.attributes.*OptionalSolo cuando exista update de perfil

6) Ejemplos practicos

Por que a veces ves JSON “gordos”: en specs y tutoriales se suele pegar un cuerpo que incluye user, session, purchase, etc. para mostrar donde van esos campos si los necesitas — no significa que cada request lleve todo. Si solo necesitas registrar un evento, basta el minimo de abajo.

0) Minimo valido (cualquier evento, p. ej. video_call_ended)

Solo meta (3 campos obligatorios) + event. Sin user, sin session, sin purchase.

{
  "meta": {
    "message_id": "018f7aa0-4f2f-7e4a-a0dc-1e6d2d69b000",
    "occurred_at": "2026-03-24T00:04:12.446Z",
    "source_system": "mi-app-o-playground"
  },
  "event": {
    "name": "video_call_ended",
    "time": "2026-03-24T00:04:12.446Z",
    "properties": {
      "consultation_id": "c_123",
      "duration_minutes": 12
    }
  }
}

7) Checklist por subsistema

  • Generar message_id unico por evento.
  • Setear occurred_at al momento real de ocurrencia.
  • Enviar solo campos relevantes; no rellenar basura.
  • No poner PII en event.properties.
  • Si reintentas, reusar el mismo message_id.

8) Atributos: guia especifica

Los atributos son estado de perfil (persistente), no eventos de actividad. Regla general: owner backend.

8.1 Que atributos van por backend vs frontend

AtributoOwner recomendadoMotivo
emailBackendPII sensible, fuente de verdad en cuenta/CRM
phoneBackendPII + formato/normalizacion + compliance
first_name, last_nameBackendConsistencia de perfil multi-sistema
dob, genderBackendDatos sensibles/regulatorios
custom.* (ej. lifecycle_stage)BackendEvitar drift semantico entre frontends
changeUser / identify anon->knownFrontend (transitorio) + Backend (final)Frontend conoce la sesion activa, backend consolida identidad

8.2 Reglas de oro para atributos

  • Actualizar solo en eventos de cambio real (no en cada page view).
  • Aplicar allowlist de atributos permitidos.
  • No sobrescribir con vacios/null salvo regla explicita.
  • Versionar cambios de semantica (lifecycle_stage_v2 si rompe compatibilidad).
  • Registrar auditoria: quien cambio, cuando, valor anterior/nuevo.

8.3 Dedupe de atributos

Para evitar writes repetidos:

  • Usar attribute_hash (hash del subset de atributos enviados).
  • Si hash no cambia, responder 202 no_change y no reenviar a Braze.
  • Guardar last_attribute_update_at por usuario.

8.6 Que NO hacer con atributos

  • No guardar metricas de sesion como atributo persistente (ej. tiempo activo por visita).
  • No mandar atributos desde multiples frontends sin normalizacion central.
  • No duplicar eventos completos en atributos.
