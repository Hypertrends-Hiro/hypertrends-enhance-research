Telemetry Catalog-First Braze Gateway

Plan para un Telemetry Genial

Objetivo: convertir telemetry en la capa central de gobierno de eventos, atributos y compras. Todo entra por el mismo API, el catalogo decide que se envia, y cada decision queda auditada.

Lo que ya esta bien

  • Ingesta unificada (/ingest y /ingest/batch).
  • Contrato universal flexible (meta/user/session/event/purchase).
  • Forward a Braze ya operativo para validar end-to-end.

Riesgos actuales

  • Idempotencia en memoria (se pierde entre procesos/restarts).
  • Catalogo aun stub (sin gobernanza real).
  • Sin outbox persistente ni retries robustos.

Regla de oro

  • Si un evento no esta en catalogo o esta deshabilitado:
  • No se reenvia a Braze.
  • Se registra rechazo estructurado con motivo.

Modelo operativo recomendado

EtapaQue haceSalida
1) Ingesta Valida payload, extrae message_id, event.name, source_system. accepted o rejected_invalid_payload
2) Catalogo Verifica si el evento existe y si esta habilitado por entorno/tenant/source. accepted_not_enabled o accepted_queued
3) Outbox Guarda item para despacho asincrono e idempotente a destinos. pending / sent / failed
4) Worker Reintenta con backoff, registra errores, mueve a dead-letter si falla. accepted_forwarded o delivery_failed_dead_letter

Plan por fases (realista y robusto)

Fase 1 - Gobernanza (1-2 semanas)

Catalogo real (JSON/DB), validacion strict de eventos, estados de respuesta estandar y logs estructurados de descarte.

Fase 2 - Persistencia (2-3 semanas)

Agregar PostgreSQL, tablas telemetry_events y telemetry_outbox, idempotencia persistente por message_id.

Fase 3 - Entrega confiable (2-3 semanas)

Worker de despacho, retries con backoff, dead-letter, metrica de entregas y errores por destino.

Fase 4 - Seguridad y operacion (1-2 semanas)

API keys por sistema emisor, rate limits, CORS por entorno, alertas de salud y tablero operativo.

KPIs clave

  • 100% mensajes con estado final trazable.
  • 0 perdidas silenciosas.
  • p95 ingesta < 150 ms (sin esperar Braze).
  • Error de delivery < 1% despues de retries.

Politica de rechazo

  • unknown_event: no existe en catalogo.
  • disabled_event: existe pero no habilitado.
  • invalid_schema: payload fuera de contrato.
  • Todos con message_id + motivo + source.
