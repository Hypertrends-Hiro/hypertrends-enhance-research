Guía de uso — Backend / workers / scripts

Plan API Payload universal (detalle) Guía Frontend Split eventos (md)

Los servicios backend, crons y scripts envían el mismo payload universal a ingest. meta.source_channel típico: backend, cron o worker.

Qué debes enviar siempre

  • meta.message_id — único por operación lógica (mismo id si reintentas el mismo job).
  • meta.occurred_at — hora de negocio (ej. hora del pedido en DB, no hora del cron).
  • meta.source_system — identificador del servicio (ej. magento-kwt-braze-sync).
  • event.name + event.time alineados al evento de dominio.
  • user.external_id cuando el usuario ya está identificado en tu sistema.

Sesión en backend

session.* suele ser opcional. Úsala si propagas session_id desde un contexto web o si correlacionas con logs.

Eventos de negocio y compras

TipoAcción

Purchase Brazepurchase.enabled=true + product_id, currency, price, quantity + event coherente (ej. purchase)
Atributos de usuariouser.email, user.attributes — ver atributos en payload-usage

Script / adapter en otro lenguaje

Puedes mantener el formato “legacy” interno y transformar a payload universal antes del POST a ingest. Así conectas sin refactor masivo del origen.

  • Mapear identidad → user.external_id
  • Generar message_id estable por fila de job (ej. hash de tenant + order_id + event_name)
  • occurred_at desde timestamp de la entidad de negocio

Ejemplo mínimo (orden confirmado)

{
  "meta": {
    "message_id": "kwt-order-1000123-purchase",
    "occurred_at": "2026-03-23T18:00:00Z",
    "source_system": "order-service",
    "source_channel": "backend",
    "tenant": "kwt",
    "schema_version": "1.0.0"
  },
  "user": { "external_id": "magento_456" },
  "event": {
    "name": "purchase",
    "time": "2026-03-23T18:00:00Z",
    "properties": { "order_id": "1000123" }
  },
  "purchase": {
    "enabled": true,
    "product_id": "SKU-1",
    "currency": "USD",
    "price": 99.9,
    "quantity": 1,
    "properties": { "order_id": "1000123" }
  }
}
