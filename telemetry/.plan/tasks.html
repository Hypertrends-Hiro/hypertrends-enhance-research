Plan de tareas — Telemetry

Tareas atómicas para el plan catalog-system-config-plan.html. Administración vía API/Postman (sin front).

Epic B — API de administración (Postman)

T-B1 POST /api/v1/catalog/events
  • Body: canonical_name (PascalCase validado), display_name, description, aliases[] opcional, lifecycle_status opcional.
  • Insert en catalog_events + aliases en transacción; 409 si canónico duplicado.
  • Auth: API key o Bearer en header (mismo mecanismo que T-B5).

Depende de: definición de auth admin (T-B5 puede hacerse primero).

T-B2 GET /api/v1/catalog/events y GET …/{canonical_name}
  • Listado paginado + filtro por lifecycle_status.
  • Detalle incluye aliases y metadata.

Depende de: T-B5.

T-B3 POST /api/v1/system-catalog-configs
  • Body: system_id, tenant, canonical_event_name (o event_id), enabled, destinations JSON (defaults Braze on / GA4 off si omitidos).
  • Resolver event_id por nombre canónico; 409 si triple duplicado.

Depende de: T-B5.

T-B4 PATCH /api/v1/system-catalog-configs/{config_id}
  • Actualizar enabled, destinations (merge o reemplazo documentado).
  • GET con filtros system_id, tenant, canonical_name opcional.

Depende de: T-B3.

T-B5 Auth + auditoría en rutas admin
  • Middleware: rechazar sin credencial en /api/v1/catalog/* (excepto públicos explícitos) y /api/v1/system-catalog-configs/*.
  • Tabla o log estructurado: actor, ruta, body resumido, antes/después en PATCH.
  • Variables: TELEMETRY_ADMIN_API_KEYS o integración OIDC (decisión única).
T-B6 OpenAPI + colección Postman exportada
  • Fragmento OpenAPI o archivo en telemetry/docs/openapi-admin.yaml.
  • Colección Postman con variables base_url, admin_key.

Depende de: T-B1–T-B4.

Epic C — Ingest y política

T-C1 Alinear respuesta ingest con estados del plan
  • Extender modelo de respuesta: status explícito (accepted, duplicate_ignored, accepted_not_forwarded) donde aplique.
  • Campo opcional destinations en respuesta con resultado por destino (texto corto).

Depende de: acuerdo con clientes del ingest.

T-C2 Persistir audit trail de ingest en Postgres
  • Tabla telemetry_ingest_audit (o nombre acordado): message_id, nombres, decisión, notas Braze, timestamps.
  • Escribir en la misma request async (o fire-and-forget con cuidado de errores).

Depende de: T-A1 si queréis migración única.

T-C3 Cache de config en memoria / Redis
  • Invalidar al recibir PATCH/POST admin o TTL corto (ej. 30s) en MVP.

Depende de: volumen esperado; opcional en dev.

T-C4 POST /api/v1/catalog/resolve (debug)
  • Input: system_id, tenant, event_name.
  • Output: canónico, event_id, fila de config resumida, si Braze/GA4 quedarían habilitados.

Depende de: T-B5.

Epic D — GA4 Measurement Protocol

T-D1 Cliente GA4 MP + mapper ga4_mp_v1
  • Leer measurement_id y resolver api_secret desde api_secret_ref → env.
  • Payload sparse: eventos + user_properties cuando corresponda; compras como evento purchase.

Depende de: T-C1 destinos en respuesta (recomendado).

T-D2 Dispatcher: respetar destinations.ga4.enabled
  • Paralelo a Braze tras pasar catálogo; errores GA4 no deben tumbar respuesta 200 del ingest (misma filosofía que Braze hoy).

Depende de: T-D1.

Epic E — Resiliencia

T-E1 Outbox + worker asíncrono
  • Tabla outbox por destino; reintentos con backoff; dead-letter.
  • Ingest solo encola o marca pending; worker envía a Braze/GA4.

Depende de: T-D2 si GA4 ya está; puede empezar solo Braze.

T-E2 Idempotencia persistente de message_id
  • Reemplazar LRU en memoria por unique constraint en Postgres (o Redis) para multi-worker.

Depende de: T-A1 / migración.

Epic F — Operación y calidad

T-F1 Tests de integración ingest + Postgres
  • pytest + contenedor Postgres o testcontainers: catálogo miss, config off, Braze mockeado (httpx mock).
T-F2 Métricas y alertas
  • Contadores: aceptados, omitidos por catálogo, errores Braze/GA4, latencia p95.
